MeWork: PRD & Specification tài liệu kiến trúc

Phiên bản: v1.0.0 Ngày cập nhật: 15/06/2026 Tác giả: Duong Nguyen Dai

1. Giới thiệu dự án

MeWork là một hệ thống runtime daemon và provider gateway tích hợp trí tuệ nhân tạo (AI-agent runtime). Hệ thống cho phép tự động hóa quy trình phát triển phần mềm bằng cách kết nối các nền tảng quản lý công việc (như bảng Kanban Mello, Jira, Linear, GitHub Issues) trực tiếp tới các bộ công cụ AI agent chạy cục bộ tại máy trạm của lập trình viên (như Claude Code CLI, Codex, OpenCode).

Thay vì sử dụng các dịch vụ AI tập trung trên đám mây gây rủi ro lộ mã nguồn và khóa API bí mật, MeWork sử dụng mô hình lai (hybrid): Máy chủ trung tâm đóng vai trò định tuyến và quản lý hàng đợi tác vụ một cách độc lập với nền tảng quản lý công việc (provider-agnostic), còn các tác vụ AI thực tế được kéo về chạy trực tiếp trên máy trạm cá nhân thông qua Daemon cục bộ, sử dụng quyền truy cập mã nguồn cục bộ an toàn.

Giá trị cốt lõi & Nguyên lý Thiết kế

Các nguyên tắc thiết kế tối thượng

Kiến trúc hệ thống được tuân thủ nghiêm ngặt theo 3 nguyên lý cốt lõi của kỹ nghệ phần mềm:

  • KISS (Keep It Simple, Stupid): Đơn giản hóa luồng giao tiếp. Máy trạm không cần cài đặt các cấu hình kết nối MCP phức tạp hay cấu hình webhook trực tiếp; máy chủ gánh vác toàn bộ trách nhiệm điều hợp.
  • YAGNI (You Aren't Gonna Need It): Không phát triển quá mức các tính năng phức tạp như hệ thống quản lý kéo thả (workflow engine DSL) hay bảng điều khiển UI phức tạp khi chưa cần thiết. Tập trung hoàn thành cơ chế xếp hàng tác vụ (queue) và bảo mật trước.
  • DRY (Don't Repeat Yourself): Gom toàn bộ logic liên lạc với các hệ thống đích (WriteBack) vào máy chủ trung tâm. Daemon máy trạm hoàn toàn không chứa bất kỳ mã nguồn chuyên biệt nào dành riêng cho Mello hay GitHub, chỉ tương tác thông qua giao thức Job chuẩn hóa.

2. Tài liệu yêu cầu sản phẩm (PRD)

Các Use Cases & Tác vụ Người dùng chính

Tác vụ 1: Kích hoạt Agent từ Kanban Board

Lập trình viên muốn gọi AI hỗ trợ sửa lỗi hoặc triển khai tính năng mới trên một ticket chỉ bằng cách viết bình luận.

  • Người dùng vào Mello, mở ticket cụ thể và để lại bình luận: /run macbook-pro sửa lỗi tràn bộ nhớ trong hàm router.go.
  • MeWork Server nhận webhook, chuyển đổi payload thành canonical job và lưu vào hàng đợi.
  • Daemon chạy trên máy macbook-pro kéo job về, gọi claude CLI sửa mã nguồn cục bộ, chạy test xác thực.
  • Daemon gửi trả kết quả (Thành công/Thất bại kèm theo logs) lên Server.
  • Server tự động ghi ngược phản hồi lên ticket Mello để người dùng theo dõi.

Tác vụ 2: Đăng ký Agent Runtime độc lập

Người dùng muốn liên kết nhiều thiết bị (PC công ty, Laptop cá nhân, VPS kiểm thử) vào tài khoản để nhận các tác vụ phù hợp.

  • Người dùng dùng CLI gõ lệnh mework runtime add --code [id] --label [name].
  • Hệ thống tạo và trả về khóa xác thực duy nhất rt_token.
  • Người dùng dùng token này khởi động daemon cục bộ mà không cần để lộ tài khoản hay mã cá nhân (PAT) trên thiết bị đó.

Yêu cầu Chức năng (Functional Requirements)

Mã yêu cầu Thành phần Mô tả chi tiết tính năng Mức độ ưu tiên
REQ-F01 Central Server Bộ định tuyến Webhook đa nền tảng (Inbound Adapter): Tiếp nhận webhook từ Mello/GitHub, xác thực chữ ký số (HMAC) và trích xuất thông tin thành CanonicalEvent chuẩn hóa. Cao (P1)
REQ-F02 Central Server Hàng đợi Tác vụ Bền vững (Job Queue): Quản lý trạng thái Job (queued, claimed, running, done, failed). Hỗ trợ cơ chế khóa (advisory lock) giúp loại bỏ xung đột tranh chấp job giữa các luồng daemon. Cao (P1)
REQ-F03 Central Server Bộ điều phối Ghi ngược (Outbound Adapter - WriteBack): Sử dụng cơ chế hộp thư gửi đi bền vững (Durable Outbox) để thực hiện kết nối ngược lại các nền tảng đích thông qua giao thức MCP (Model Context Protocol). Tự động thử lại với lũy thừa thời gian nếu lỗi kết nối. Cao (P1)
REQ-F04 Local Daemon Trình tiêu thụ hàng đợi (Long-Polling Client): Thực hiện kết nối long-polling wait=25s lên server bằng rt_token để nhận việc ngay lập tức mà không tiêu tốn tài nguyên tải lặp. Cao (P1)
REQ-F05 Local Daemon Bộ thực thi AI cục bộ (Agent Process Runner): Phát hiện môi trường (claude, codex, opencode) trong hệ thống. Khởi chạy AI bằng đường ống stdin biệt lập, tránh lỗi command injection, kiểm soát thời gian chạy (timeout). Cao (P1)
REQ-F06 Local Daemon Giữ nhịp kết nối (Heartbeat): Chạy một tiến trình ngầm gửi heartbeat định kỳ 30 giây lên server trong khi AI đang chạy cục bộ để gia hạn thời gian chiếm giữ Job (lease), ngăn ngừa việc Job bị thu hồi do chạy quá lâu. Cao (P1)
REQ-F07 CLI Command Quản lý tiến trình Daemon cục bộ: Các lệnh kiểm soát daemon bao gồm: start (tách tiến trình nền), stop (dừng an toàn bằng tín hiệu), status, restart, logs. Trung bình (P2)

Yêu cầu Phi chức năng (Non-Functional Requirements)

1. An toàn và Bảo mật (Security)

  • Bảo mật thông tin đăng nhập: Daemon máy trạm không lưu trữ Mello Personal Access Token (PAT). Thay vào đó, máy trạm chỉ sử dụng rt_token (sinh ra từ cơ chế băm HMAC một chiều) có quyền hạn bị giới hạn chỉ để nhận và báo cáo các Job thuộc về runtime đó.
  • Mã hóa dữ liệu nhạy cảm: Mọi token dịch vụ liên kết (ví dụ: thông tin kết nối MCP) lưu trên máy chủ trung tâm phải được mã hóa đối xứng AES-256-GCM ở tầng cơ sở dữ liệu (sử dụng khóa bí mật MEWORK_SECRET_KEY nạp từ biến môi trường).
  • Bảo vệ thực thi mã nguồn: Đầu vào của AI Agent được đẩy hoàn toàn thông qua đường ống luồng dữ liệu chuẩn (Standard Input - stdin), ngăn chặn tuyệt đối các cuộc tấn công chèn lệnh hệ thống thông qua tham số dòng lệnh (Command Injection).

2. Tính tin cậy và Khả năng chịu lỗi (Reliability)

  • Chống trùng lặp tác vụ (Idempotency): Tận dụng ràng buộc duy nhất composite index của PostgreSQL trên cột (provider_code, external_event_id) để đảm bảo mỗi hành động kích hoạt trên bảng Kanban chỉ tạo ra duy nhất một Job trong hàng đợi, ngay cả khi webhook bị gửi lặp lại nhiều lần.
  • Thu hồi Job chết: Khi một máy trạm bị mất điện hoặc mất mạng đột ngột lúc đang chạy AI, cơ chế hết hạn thời gian thuê (claim lease expire) trên server sẽ tự động phát hiện và giải phóng Job trở lại hàng đợi sau khi hết chu kỳ heartbeat, đảm bảo hệ thống không bị treo.

3. Hiệu năng (Performance)

  • Độ trễ phản hồi thấp: Sử dụng phương thức HTTP Long-Polling giúp daemon nhận việc ngay lập tức trong vòng dưới 100ms kể từ khi Job được đẩy vào hàng đợi, thay vì phải chạy vòng lặp truy vấn liên tục gây quá tải cơ sở dữ liệu.
  • Truy vấn tối ưu: Tận dụng chỉ mục hỗn hợp (Partial Index) trên cơ sở dữ liệu cho các tác vụ lấy Job idx_jobs_claim ON jobs (runtime_id, status, created_at) và xử lý outbox idx_jobs_writeback ON jobs (writeback_status) WHERE writeback_status = 'pending'.

Kế hoạch Mở rộng trong tương lai

  • Tích hợp các Provider phổ biến khác như: GitHub Issues, Linear, Jira Cloud bằng cách bổ sung thêm cấu trúc adapter kế thừa interface Provider có sẵn trên server mà không cần sửa đổi mã nguồn của Daemon.
  • Cơ chế gom luồng thực thi song song trên máy trạm để xử lý nhiều yêu cầu đồng thời từ bảng Kanban (Multi-job concurrency).
  • Xây dựng cơ chế cập nhật tự động an toàn cho daemon CLI thông qua lệnh mework update khi hệ thống phát hành phiên bản mới trên GitHub.

3. Thiết kế Kiến trúc & Mã nguồn

Mô hình Hệ thống & Ranh giới trách nhiệm

Hệ thống được thiết kế theo kiến trúc hướng cổng và bộ điều hợp (Hexagonal / Ports-and-Adapters). Toàn bộ kiến trúc được phân tách thành 3 vùng ranh giới rõ rệt:

  1. Không gian Máy trạm (Workspace Client): Nơi chứa mã nguồn của dự án cần phát triển, daemon cục bộ thực thi tác vụ AI thông qua môi trường cách ly (Sandbox directory).
  2. Máy chủ Trung tâm (MeWork Server): Nhận và phân tích webhook, quản lý hàng đợi và trạng thái Job, mã hóa thông tin bảo mật, xử lý outbox ghi ngược kết quả thông qua MCP Client.
  3. Nền tảng Quản lý công việc ngoại vi (Task System Platform): Nơi sinh ra sự kiện kích hoạt tác vụ (Kanban Board) và tiếp nhận bình luận báo cáo kết quả của AI.

Lưu ý về trạng thái triển khai (Implementation Status)

Đây là thiết kế kiến trúc mục tiêu (target) của hệ thống. Tại thời điểm hiện tại, dự án đang trong giai đoạn Pre-Release: phần lớn các thành phần đang ở trạng thái thiết kế hoặc đang phát triển. Chỉ tầng schema cơ sở dữ liệu (Phase 01) đã hoàn thành. Bảng dưới đây phản ánh đúng trạng thái mã nguồn hiện tại để tránh hiểu nhầm.

Sơ đồ Kiến trúc Hệ thống (Architecture Diagram)

Luồng tuần tự chi tiết (Sequence Flows)

1
Gửi Webhook & Xác thực Đầu vào

Webhook được bắn từ bảng Kanban Mello tới MeWork Server. Ở tầng Inbound Adapter, server xác thực tính hợp lệ bằng khóa bí mật HMAC cấu hình trong bảng provider_connections. Nếu thành công và phát hiện đúng cú pháp trigger /run, server trích xuất dữ liệu thành thực thể CanonicalEvent.

2
Xếp hàng & Giải quyết tranh chấp (Advisory Lock)

CanonicalEvent được nạp các thông tin bổ sung từ Profile (bao gồm chỉ dẫn prompt gốc và cấu hình workflow) rồi lưu vào cơ sở dữ liệu Postgres ở trạng thái queued. Ràng buộc composite unique giúp lọc trùng lặp webhook gửi lỗi.

3
Báo nhận & Cho thuê Job (Lease Claim)

Daemon máy trạm thực hiện long-poll /v1/jobs/next. Server kiểm tra danh tính runtime và áp dụng khóa Advisory Lock trên dòng của PostgreSQL để cập nhật trạng thái Job từ queued sang claimed và gán nhãn thời gian thuê (lease) giới hạn. Nếu có nhiều Daemon cùng poll một lúc, cơ chế đảm bảo chỉ có duy nhất một Daemon nhận được Job.

4
Thực thi Sandbox & Cập nhật nhịp tim

Daemon cục bộ tải Job về, tạo thư mục cách ly ~/.mework/work/[job-id], khởi chạy AI Agent CLI bằng luồng dữ liệu stdin. Trong quá trình chạy, một luồng phụ liên tục gửi heartbeat lên server để gia hạn quyền chiếm giữ Job (lease), ngăn ngừa các máy trạm khác cướp Job.

5
Báo cáo kết quả & Ghi ngược (Outbox)

Khi AI chạy xong (hoặc thất bại), Daemon gửi báo cáo ACK kèm trạng thái (done hoặc failed) và tóm tắt kết quả về Server. Server ghi nhận trạng thái Job đã hoàn thành, đồng thời tạo một bản ghi outbox ghi ngược kết quả (WriteBack). Tiến trình phụ (Outbox Worker) trên server sẽ tự động kết nối tới MCP Server ngoại vi để viết bình luận phản hồi kết quả lên Kanban board.

Thiết kế Cơ sở dữ liệu (Database Schema)

Cơ sở dữ liệu sử dụng PostgreSQL để hỗ trợ cơ chế lưu trữ hàng đợi tác vụ bền vững, kiểm soát trạng thái tranh chấp đồng thời và mã hóa bí mật kết nối dịch vụ ngoại vi. Dưới đây là cấu trúc các bảng dữ liệu chính (được viết trong Goose migration 000001_init.sql):

1. Bảng accounts (Tài khoản người dùng)

Lưu trữ thực thể tài khoản chính sở hữu các kết nối và runtimes.

Tên trường Kiểu dữ liệu Mô tả / Ràng buộc
id UUID PRIMARY KEY, DEFAULT gen_random_uuid()
name VARCHAR(255) Tên người dùng (không được NULL)
created_at TIMESTAMPTZ Thời gian tạo tài khoản

2. Bảng provider_connections (Kết nối dịch vụ ngoại vi)

Lưu thông tin khóa webhook, URL MCP của từng nhà cung cấp nhiệm vụ tương ứng với tài khoản người dùng.

Tên trường Kiểu dữ liệu Mô tả / Ràng buộc
id UUID PRIMARY KEY
account_id UUID FOREIGN KEY references accounts(id) ON DELETE CASCADE
provider_code VARCHAR(255) Mã nhà cung cấp (ví dụ: 'mello', 'github', 'jira')
webhook_secret TEXT Khóa xác thực chữ ký số webhook
mcp_url TEXT Địa chỉ endpoint MCP dùng để ghi ngược (WriteBack)
mcp_auth_enc TEXT Khóa xác thực MCP đã được mã hóa đối xứng AES-256-GCM
config JSONB Cấu hình bổ sung riêng của Provider (mặc định '{}')
UNIQUE (account_id, provider_code) Constraint Mỗi tài khoản chỉ kết nối tối đa 1 instance của một loại Provider

3. Bảng account_identities (Danh tính người dùng theo Provider)

Ánh xạ ID người dùng trên hệ thống ngoại vi về tài khoản MeWork, phục vụ xác thực commenter (authz) trước khi tạo Job.

Tên trường Kiểu dữ liệu Mô tả / Ràng buộc
account_id UUID FOREIGN KEY references accounts(id) ON DELETE CASCADE
provider_code VARCHAR(255) Mã nhà cung cấp ('mello', 'github', ...)
external_user_id VARCHAR(255) ID người dùng trên hệ thống ngoại vi
PRIMARY KEY (account_id, provider_code) Constraint Mỗi tài khoản có 1 danh tính trên mỗi Provider
UNIQUE (provider_code, external_user_id) Constraint Tra cứu ngược: ID ngoại vi → account (chống mạo danh)

4. Bảng watched_containers (Container được giám sát)

Ánh xạ board/project/repo trên hệ thống ngoại vi về tài khoản. Cốt lõi để webhook resolve được Job thuộc account nào (fail-closed nếu không khớp).

Tên trường Kiểu dữ liệu Mô tả / Ràng buộc
account_id UUID FOREIGN KEY references accounts(id) ON DELETE CASCADE
provider_code VARCHAR(255) Mã nhà cung cấp
external_container_id VARCHAR(255) ID của board/project/repo trên hệ thống ngoại vi
PRIMARY KEY (account_id, provider_code, external_container_id) Constraint Khóa chính tổng hợp 3 cột
UNIQUE (provider_code, external_container_id) Constraint Một container chỉ thuộc duy nhất 1 account (resolve webhook → account)

5. Bảng runtimes (Đại diện máy trạm Agent)

Quản lý danh sách các thiết bị hoặc máy trạm chạy Daemon đã đăng ký nhận việc.

Tên trường Kiểu dữ liệu Mô tả / Ràng buộc
id UUID PRIMARY KEY
account_id UUID FOREIGN KEY
code VARCHAR(255) Mã runtime do người dùng đặt (ví dụ: 'macbook-pro')
token_lookup VARCHAR(255) Băm một chiều của rt_token phục vụ tìm kiếm nhanh O(1) [UNIQUE]
status VARCHAR(50) Trạng thái (online, offline)
UNIQUE (account_id, code) Constraint Mỗi tài khoản không được trùng mã runtime

6. Bảng profiles (Cấu hình chỉ dẫn & Workflow)

Lưu trữ các bộ prompt nghiệp vụ (system prompt) cùng cấu hình harness/workflow trên server, được snapshot vào Job tại thời điểm enqueue.

Tên trường Kiểu dữ liệu Mô tả / Ràng buộc
id UUID PRIMARY KEY
account_id UUID FOREIGN KEY references accounts(id) ON DELETE CASCADE
name VARCHAR(255) Tên profile do người dùng đặt
body TEXT Nội dung prompt nghiệp vụ (markdown)
backend_hint VARCHAR(255) Gợi ý backend AI ưu tiên (có thể NULL)
harness VARCHAR(255) Loại harness mục tiêu (ví dụ: 'claude-code')
workflow_config JSONB Cấu hình workflow (allowed tools, template…), mặc định '{}'
UNIQUE (account_id, name) Constraint Mỗi tài khoản không được trùng tên profile

7. Bảng jobs (Hàng đợi công việc chi tiết)

Nơi lưu giữ trạng thái, nội dung ticket, kết quả thực thi và lịch sử ghi ngược.

Tên trường Kiểu dữ liệu Mô tả / Ràng buộc
id UUID PRIMARY KEY
runtime_id UUID FOREIGN KEY references runtimes(id)
external_task_id VARCHAR(255) ID của ticket trên hệ thống ngoại vi (ví dụ: ticket Mello)
external_event_id VARCHAR(255) ID của comment/sự kiện trigger kích hoạt (Idempotency Key)
status VARCHAR(50) Trạng thái Job: queued, claimed, running, done, failed
writeback_status VARCHAR(50) Trạng thái outbox: pending, processing, done, failed
instructions TEXT Nội dung chỉ dẫn người dùng viết sau từ khóa trigger
claim_lease_until TIMESTAMPTZ Thời hạn cho thuê Job (được gia hạn liên tục bằng Heartbeat)
UNIQUE (provider_code, external_event_id) Constraint Ngăn cản tạo trùng Job khi webhook gửi lại nhiều lần

Các chỉ mục tối ưu hiệu năng (Database Indexes)

-- Chỉ mục phục vụ truy vấn kéo tác vụ cực nhanh (claim query)
CREATE INDEX idx_jobs_claim ON jobs (runtime_id, status, created_at);

-- Ràng buộc mềm ngăn chặn một runtime chạy 2 jobs song song đồng thời
CREATE UNIQUE INDEX idx_jobs_one_active_per_runtime ON jobs (runtime_id)
WHERE status IN ('claimed', 'running');

-- Chỉ mục lọc outbox phục vụ worker ghi ngược
CREATE INDEX idx_jobs_writeback ON jobs (writeback_status)
WHERE writeback_status = 'pending';

Cơ chế Bảo mật (Security Implementations)

Cơ chế Mã hóa đối xứng AES-256-GCM của Secrets

Các thông tin nhạy cảm của người dùng (chẳng hạn như khóa API hoặc token kết nối MCP trong bảng provider_connections.mcp_auth_enc) được bảo mật nghiêm ngặt bằng phương pháp mã hóa đối xứng AES-256-GCM.

Khóa mã hóa MEWORK_SECRET_KEY là khóa 32-byte được nạp từ biến môi trường của máy chủ. Khi ghi thông tin kết nối, máy chủ sẽ sinh ra một chuỗi ngẫu nhiên nonce (12-byte), thực hiện mã hóa thông tin và ghép nonce ở đầu bản mã trước khi lưu trữ vào DB. Tiến trình WriteBack giải mã bản mã này trước khi tạo kết nối tới MCP Server ngoại vi.

4. Hướng dẫn & Vận hành

CLI Command Reference (Lệnh điều khiển mework)

Công cụ CLI mework hỗ trợ ba nhóm lệnh chính phục vụ phát triển, vận hành và cấu hình hệ thống:

1. Nhóm lệnh lõi (Core Commands)

  • mework login --token <token>: Xác thực tài khoản của bạn với hệ thống Mello bằng Personal Access Token.
  • mework logout: Đăng xuất, xóa cấu hình token cục bộ.
  • mework board list: Liệt kê danh sách các bảng công việc đang được quản lý.
  • mework ticket list --board-id <id>: Liệt kê các ticket đang có trên một bảng cụ thể.
  • mework ticket get <ticket-id>: Xem chi tiết nội dung mô tả, tiêu đề và bình luận của ticket.

2. Nhóm lệnh Runtime & Daemon (Từ Phase 2 trở đi)

  • mework runtime add --code <code> --label <name>: Đăng ký một thực thể máy trạm và nhận khóa rt_token.
  • mework daemon start [--foreground]: Khởi chạy Daemon xử lý tác vụ ngầm (hoặc chạy nổi hiển thị log).
  • mework daemon status: Kiểm tra trạng thái hoạt động của daemon cục bộ (hiển thị PID và cổng Health).
  • mework daemon stop: Dừng daemon một cách an toàn thông qua cổng Health nội bộ.
  • mework daemon logs [-f]: Đọc và theo dõi tệp nhật ký ghi chép của daemon tại máy trạm.

3. Nhóm lệnh Quản lý Profile (Từ Phase 4 trở đi)

  • mework profile add --name <name> --file <path>: Tải lên cấu hình profile (prompt nghiệp vụ) từ file markdown lên server trung tâm để sẵn sàng nhúng vào Job.
  • mework profile list: Danh sách các profile có trên tài khoản.

Cấu hình Môi trường (Environment Variables)

Để vận hành máy chủ MeWork Server (mework-server), cần nạp đầy đủ các biến môi trường sau đây: 

# Địa chỉ kết nối cơ sở dữ liệu PostgreSQL
DATABASE_URL=postgres://user:password@localhost:5432/mework_db?sslmode=disable

# Địa chỉ và Cổng lắng nghe HTTP của Server
LISTEN_ADDR=0.0.0.0:8080

# Khóa bí mật dùng để mã hóa thông tin kết nối MCP (yêu cầu khóa base64 32-byte)
MEWORK_SECRET_KEY=y3S1eWN1cmVTZWNyZXRLZXlGb3JNZVdvcmtBZXMyNTZHQ00=
